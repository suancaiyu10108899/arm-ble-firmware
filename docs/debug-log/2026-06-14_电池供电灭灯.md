# Debug Log: 电池供电时板子蓝灯不亮、手柄连不上

> **日期**: 2026-06-14 22:48
> **固件**: v2.7
> **发现人**: 手动排查

## 现象

MicroUSB 供电 → 板子正常启动 → 蓝灯亮 → 自动扫描连接手柄 ✅
拔掉 MicroUSB → 用电池供电 → 蓝灯闪一下灭 → 手柄蓝灯持续闪烁（没连上）

## 排查过程

1. 怀疑供电不足 → 量电压 3.269V → 发现 BAT 引脚接 3.3V 不够（需 4V+）→ 排除
2. 用 5V 接 BAT → 仍灭灯 → 不是供电问题
3. MicroUSB 重新插上、不开串口监视器 → 板子蓝灯也不亮 → 定位到串口相关
4. 检查 `setup()` 代码 → 发现 `while (!Serial)` 死等

## 根因

```cpp
// main.cpp:142 (修复前)
Serial.begin(115200);
while (!Serial) delay(10);
```

nRF52840 使用原生 USB CDC。当没有 USB Host（电脑）连接时，`Serial` 对象永远不可用，`!Serial` 永远返回 `true`。板子死在 `while (!Serial)` 循环中，`setup()` 后面的 BLE 初始化、手柄扫描全部无法执行。

MicroUSB 供电时有电脑连接 → `Serial` 立即可用 → 不卡。电池供电或 MicroUSB 只供电不接电脑 → 永远卡在 setup 开头。

## 修复

```cpp
// main.cpp:141-142 (修复后)
Serial.begin(115200);
unsigned long usbTimeout = millis() + 3000;
while (!Serial && millis() < usbTimeout) delay(10);
```

3 秒超时：USB 连接时正常等待串口打开，电池供电时 3 秒后自动跳过，继续 BLE 初始化。

## 教训

- 手写/读懂每一行代码是调试的前提。这个 bug 写代码时 AI 没注意，排查时也是靠自己看代码才定位到。
- AI 辅助开发可以提效，但物理直觉（"蓝灯不亮 → 板子没启动"）和独立排查能力不可替代。
- nRF52840 原生 USB 的特性：`!Serial` 在无 Host 时永远为 true。这是一个平台相关的坑，依赖文档和经验。
